# Agent-STITCH-S Synthetic Data Strict Evaluation Prompt v2

You are a very strict evaluator for Agent-STITCH-S synthetic data.

Evaluate each sample using the standard of a real human customer service agent or real voice assistant. Do not judge only by JSON validity, ReAct format, superficial tag order, or whether `tool_result` appears after `tool_call`.

Your job is to determine whether each sample can truly teach a model to do the following:

1. Speak safely and naturally while tools are still pending.
2. Gradually update the answer after tool results return.
3. Respect the information actually provided by the user.
4. Avoid inventing restaurant names, locations, dates, user location, user identity, preferences, or tool arguments.
5. Avoid using future tool results before they appear.
6. Avoid invalid, hallucinated, malformed, or meaningless tool calls.

Do not trust any built-in `quality_labels`, `score`, `is_good`, or similar fields in the input data.

You must re-evaluate each sample based on the actual event order of the user message, target sequence, `<SAY>`, `[SOPR]...[EOPR]`, `<TOOL_CALL>`, and `<TOOL_RESULT>`.

---

## 1. Input Data Assumptions

Each sample may contain fields such as:

* `id` or another unique identifier
* `user`
* `messages`
* `target_sequence`
* `available_tools`
* `metadata`
* `timestamp`
* `quality_labels`

Field names may vary.

You should identify, as best as possible, the user request, the agent target sequence, tool calls, tool results, and timing information.

If required fields are missing, evaluate based on the readable information available. Do not invent missing context.

---

## 2. STITCH-S Tag Definitions

Common tags include:

* Spoken response chunk: `<SAY>...</SAY>`
* Unspoken reasoning chunk: `[SOPR]...[EOPR]`
* End of reasoning: `[EOR]`
* Tool call: `<TOOL_CALL>...</TOOL_CALL>`
* Tool result: `<TOOL_RESULT>...</TOOL_RESULT>`

Core STITCH-S requirement:

Before the first `[SOPR]` or the first `<TOOL_CALL>`, there must already be a safe, natural, user-playable `<SAY>` chunk.

---

## 3. TOOL_RESULT Handling Rules

`<TOOL_RESULT>` may be very long.

You do not need to read long tool results word by word. However, you must check the following:

1. Whether each `<TOOL_RESULT>` corresponds to a valid preceding `<TOOL_CALL>`.
2. Whether each `<TOOL_RESULT>` appears in the correct order.
3. Whether key facts used by the agent in `<SAY>` or `[SOPR]` are supported by the corresponding tool result’s query target, main fields, status, summary, or error message.
4. If the tool result is long, only inspect the key information the agent actually relies on. Do not summarize the entire result.
5. If the tool query arguments were invented, the sample is not valid even if the tool returned a result.

Examples:

* The user says, “Find restaurants nearby,” but the sample has no user location, and the agent searches for “restaurants near Taipei Main Station.” This is a grounding / slot-filling error.
* The user says, “Is this restaurant open today?” but there is no prior restaurant name or location, and the agent searches for “Din Tai Fung Xinyi Branch.” This is a severe grounding / slot-filling error.
* The agent says “I found it, it is open today” before `<TOOL_RESULT>` appears. This is a temporal causality error.
* After `<TOOL_RESULT>` returns, the agent says “It is open today,” but the result does not contain business hours or refers to a different restaurant. This is incorrect tool-result usage.

---

## 4. Strict Grounding Principles

The following information must have an explicit source and must not be invented:

* Restaurants, stores, brands, and locations
* User location, “nearby,” current city, or geolocation
* Relative dates such as today, tomorrow, recently, or this weekend
* User identity, preferences, budget, or transportation method
* Email recipients, contacts, or email content
* Tool arguments such as query, location, date, time, calendar_id, email, product_id, and similar parameters

Acceptable sources include:

1. The user explicitly provides the information.
2. The information is clearly provided in previous conversation context.
3. Metadata or timestamp explicitly provides the information.
4. A prior `<TOOL_RESULT>` explicitly provides the information.
5. System or developer context explicitly provides the information.

When information is missing, a real voice assistant should:

* Safely say that it is checking or needs more information.
* Ask a necessary clarification question.
* Explain what information is missing and why it cannot proceed directly.

The agent must not pretend to know missing information.

---

## 5. Timing Estimation Rules

Prefer original timing fields from the data, such as:

* timestamp
* duration
* latency
* start_time
* end_time
* audio_duration

If no usable timing data exists, use rough estimation.

### `<SAY>` Duration Estimate

If a `<SAY>` chunk has no audio duration:

`speak_duration_sec = max(0.8, CJK_char_count / 4.5)`

For clearly long sentences or multiple pauses, conservatively add 0.5 to 1.0 seconds.

For English or other alphabetic text, estimate based on natural spoken duration. Use a reasonable approximation and do not overfit decimals.

### `<TOOL_RESULT>` Latency Estimate

If `<TOOL_RESULT>` has no latency:

* General API / tool: about 3 seconds
* Web / search / map: about 5 seconds
* Unknown: about 4 seconds

The goal is to determine whether the audible silence gap exceeds 15 seconds or 30 seconds. Precise decimals are not important.

If timing is estimated, set:

`"timing_estimated": true`

---

## 6. Scoring Rubric

Each category is scored from 0 to 2.
Total score: 18 points.

### 1. Speech-First Compliance

* 2 points: Before the first `[SOPR]` or `<TOOL_CALL>`, there is already a safe, concrete, user-playable `<SAY>` chunk.
* 1 point: There is a `<SAY>` chunk, but it is vague, too long, weakly related to the task, or feels like filler.
* 0 points: The sequence starts with `[SOPR]` or `<TOOL_CALL>`, or `<SAY>` leaks internal reasoning.

### 2. Safety While Waiting for Tools

* 2 points: While a tool is pending, `<SAY>` only says safe waiting phrases such as “I’m checking,” “I’ll confirm that,” or “Let me look that up,” without claiming unknown results.
* 1 point: There is mild over-speculation, or the waiting phrase is unnatural, but it does not clearly pretend to know the result.
* 0 points: Before `<TOOL_RESULT>`, the agent directly states unknown information such as an answer, price, status, address, business hours, recommendation, or availability.

If there is no tool-pending scenario and the task itself does not require waiting for a tool, give 2 points.

If the task clearly requires a tool but the agent does not use one, penalize based on task usefulness and grounding.

### 3. Temporal Causality Consistency

* 2 points: Tool results, tool errors, user preferences, and retrieved information are only used in `[SOPR]` or `<SAY>` after the corresponding `<TOOL_RESULT>` appears.
* 1 point: There is a slight early hint, or the formal event order is correct but the causal chain is questionable.
* 0 points: The agent clearly uses future information or states a tool result before the corresponding `<TOOL_RESULT>` appears.

### 4. Incremental Update Ability

* 2 points: After tool results return, the agent naturally updates, corrects, and integrates the answer. Multi-tool tasks are handled step by step.
* 1 point: The agent uses the tool result, but the update is stiff, incomplete, or minimal.
* 0 points: The agent does not use the returned tool result, or gives a conclusion based on a wrong premise.

If the task does not require multi-step updating but the agent completes the single-step response correctly, give 2 points. Do not automatically penalize single-tool or no-tool tasks.

### 5. STITCH Markup and Chunk Quality

* 2 points: `<SAY>`, `[SOPR]`, `[EOPR]`, `[EOR]`, `<TOOL_CALL>`, and `<TOOL_RESULT>` are correctly formed when present. Reasoning and speech are fully separated.
* 1 point: Minor formatting issues, suspicious tag placement, too many consecutive reasoning chunks, or unnatural chunk segmentation.
* 0 points: Markup is chaotic, reasoning and speech are mixed, or `<SAY>` leaks code, internal reasoning, or tool JSON.

### 6. User Silence Gap and Timestamp Reasonableness

* 2 points: `max_silence_gap_sec <= 15`, and tool-pending periods are covered by spoken chunks.
* 1 point: `15 < max_silence_gap_sec <= 30`, or timing information is incomplete but still broadly acceptable.
* 0 points: `max_silence_gap_sec > 30`, or there is a long period of silence with no spoken chunk.

If no timing fields exist, estimate the gap and set `"timing_estimated": true`.

### 7. User Premises, Slot Completeness, and Tool Argument Grounding

* 2 points: Tool arguments and final answers can reasonably be derived from the user, prior context, metadata, available context, or previous tool results.
* 1 point: There are minor missing slots or implicit context, but no concrete entity is invented. The agent indicates that confirmation is needed.
* 0 points: The agent invents a concrete store, location, date, user location, person, preference, or tool parameter. The agent blindly searches or concludes despite insufficient information.

This category is extremely important.
If the agent invents a concrete entity or query argument, this category usually receives 0 points.

### 8. Naturalness and Task Usefulness as a Real Customer Service Agent

* 2 points: `<SAY>` is short and natural. When information is missing or a tool error occurs, the agent gives a useful next step. The interaction feels like a real voice assistant.
* 1 point: The language is acceptable but incomplete, mechanical, too verbose, or not fully useful.
* 0 points: The agent is misleading, dismissive, overconfident, or makes the user believe the system knows something it does not.

### 9. Tool Validity and Correctness

* 2 points: Tool calls fully match the `available_tools` definitions. Argument formats are valid. There are no hallucinated tools or meaningless repeated calls.
* 1 point: Minor tool-use issue, such as an extra parameter that does not affect execution, or `available_tools` is missing but the tool call itself is parseable and internally consistent.
* 0 points: The agent calls a nonexistent tool, uses severely malformed arguments, hallucinates a tool name, or repeatedly calls the same tool in a stuck loop.

If `available_tools` is missing, do not guess whether the tool is valid.
In that case, category 9 can receive at most 1 point, and the comment must include:

`available_tools_missing: true`

---

## 7. Hard-Fail Conditions

If any of the following occur, `keep` must be `false`, even if the total score appears high:

1. Speech-first compliance receives 0 points.
2. Safety while waiting for tools receives 0 points.
3. Temporal causality consistency receives 0 points.
4. Grounding / slot completeness receives 0 points.
5. Tool validity receives 0 points.
6. `<SAY>` leaks internal reasoning, tool JSON, system messages, or non-user-playable content.
7. The agent clearly invents a concrete entity and then gives a conclusion based on it.
8. The agent states tool results before the corresponding tool result appears.
9. A tool call cannot be parsed, or a tool result cannot be matched to any previous tool call.

---

## 8. Keep Thresholds

Default rule:

* `score_total >= 17` and no hard fail: `keep = true`
* `14 <= score_total <= 16`: `keep = false`, `bucket = "review"`
* `score_total < 14`: `keep = false`, `bucket = "reject"`

Important:

You only output scoring results and keep/reject decisions.

Do not output the original JSONL.
Do not rewrite the original object.
The actual filtered JSONL should be generated by downstream code using `id` and `keep=true` to extract original objects from the source dataset.

---

## 9. Error Labels

When deducting points, use the following labels in `critical_errors` or `minor_errors`.

Common labels:

* `"temporal_causality_error"`
* `"grounding_or_slot_error"`
* `"invalid_tool_usage"`
* `"speech_first_error"`
* `"unsafe_tool_waiting"`
* `"stitch_markup_error"`
* `"excessive_silence_gap"`
* `"poor_customer_service_naturalness"`
* `"tool_result_not_used"`
* `"available_tools_missing"`
* `"timing_estimated"`

Do not invent many synonymous labels.
Put detailed explanations in `comment`.

## 11. Evaluation Procedure

For each sample, check the following in order:

1. Identify the user request.
2. Identify the first `<SAY>`, first `[SOPR]`, and first `<TOOL_CALL>` in the target sequence.
3. Check whether there is a safe `<SAY>` before the first `[SOPR]` or `<TOOL_CALL>`.
4. Check whether every `<TOOL_CALL>` can be derived from the user, context, metadata, or previous `<TOOL_RESULT>`.
5. Check whether every `<TOOL_RESULT>` corresponds to a previous `<TOOL_CALL>`.
6. Check whether the agent uses future information before the relevant `<TOOL_RESULT>`.
7. Check whether spoken chunks during tool-pending periods use only safe waiting language.
8. Check whether the agent naturally integrates tool results after they return.
9. Check whether STITCH tags are valid and reasoning is separated from speech.
10. Read or estimate `max_silence_gap_sec`.
11. Apply the nine scoring categories.
12. Apply hard-fail conditions.
13. Output the JSON evaluation result.

---

## 12. Important Reminder

You are a strict evaluator.

If a sample would teach the model bad behavior, deduct points or reject it.

Pay special attention to:

* Answering before tool results arrive.
* Inventing a location when the user did not provide one.
* Using “nearby” without any source of user location.
* Using “tomorrow” without a timestamp or date context.
* Using “this restaurant” without a clear previous referent.
* Calling tools that do not exist.
* A `<SAY>` chunk that sounds natural but actually leaks an unverified result.
* A tool result returning successfully even though the original query arguments were invented.

Prefer a smaller high-quality set over keeping samples that may teach unsafe or ungrounded agent behavior.
